A Note on Build Systems

Build systems are awesome. If you're developing similar websites or apps over and over using the same tools, you should look into creating a small build system that will give you a head start on setting up a project. You can always add packages or tools to the build, but you can't get back the time it takes to set up the same baseline tools you used in the last 10 projects.

One thing I've been struggling with for a few weeks is a build system that was handed to me from an outside company. I've been fiddling with it and just can't get it to work. I'm trying to deconstruct and reconstruct it now but there are some things baked into it that I'm having a hard time extracting.

Based on this experience, one question I think you need to consider when building your own system is: Can someone else easily set up and step into this workflow? Can someone who doesn't own my hardware take this entire system and use it without issue?

It's probably impossible to know if your system works on all computers, but here are some general strategies that I've thought of:

  • Use open source technologies
  • Use technologies that are cross-platform
  • Keep your abstraction layers as flat as possible
  • Put the code on Github or some other Git service
  • Write documentation and fix bugs as they happen

In my experience of sharing my own build system and using the build systems of others, these are pretty good guidelines. While you want to make your build system your own and utilize tools that help you build websites more efficiently, I believe you should also consider what would happen if you had to hand off the entire build system (not just the output files) to another developer.
